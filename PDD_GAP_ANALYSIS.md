# PDD Gap Analysis: Why the Recruiting App Had 16 Bugs

## Executive Summary

PDD generated a recruiting system from PRD #516, split into 3 sub-issues (#519 Core Backend, #520 Nurture Pipeline, #521 Frontend). All 3 sub-issues passed all 12 validation steps — yet the generated code contained **16 bugs** (#552–#567) requiring manual fixes across 15 files.

This report maps every bug to a specific failure in PDD's pipeline, identifies 7 root-cause categories, and proposes concrete fixes to PDD's prompts and orchestrator.

**Key finding**: **14 of 16 bugs originated from prompt defects.** The code generator was faithful to the prompts — the prompts themselves were wrong.

The bugs fall into two categories:
- **Prompt defects** (14 bugs): The prompts generated by `pdd arch` contained errors or critical omissions — most damagingly, a phantom camelCase serialization assumption that doesn't exist in the codebase
- **Non-prompt issues** (2 bugs): One correct prompt that wasn't applied (#561), and one LLM syntax error (#563)

**Root proposal**: PDD should generate **shared context documents** (data dictionary, API contracts, integration points) before generating per-module prompts. This single change would have prevented 14 of 16 bugs.

## 2. Bug-to-Gap Mapping

### Gap 1: No Entry Point Wiring System
**Bugs**: #552, #560, #561, #562 (4 bugs — 2×P0, 2×P1)
**Failure layer**: Architecture

PDD generates individual modules but has **no mechanism** to wire them into existing entry points:

| Bug | What happened | What PDD should have done |
|-----|--------------|--------------------------|
| #552 | 10 Cloud Functions generated but never registered in `main.py` | Generate a patch to `main.py` adding `@https_fn.on_request()` registrations |
| #560 | No index page at `/admin/recruiting/` — only sub-paths existed | Generate a redirect page or include an index route in architecture |
| #561 | Admin sidebar `layout.tsx` had no links to recruiting pages | Generate a patch to `admin/layout.tsx` adding nav items |
| #562 | `recruiting/__init__.py` got duplicate import blocks | Coordinate imports across modules; detect duplicates |

**Pipeline failure point** (`pdd/pdd/templates/architecture/architecture_json.prompt`):
- The `architecture.json` schema has no `entryPoints` field — modules are "orphan" by default
- Each module maps 1:1 to a file. There is no concept of "patch an existing file"
- Step 9 completeness check (`agentic_arch_step9_completeness_LLM.prompt`) checks if entry point files **exist** in the architecture but not that **existing** entry points in the codebase are **updated**

**Evidence from Step 9 prompt** — The completeness check includes:
> "Layer Completeness Check: Web apps need entry point, root layout, all routes, API routes, types"

This checks that the architecture **defines** these items, but doesn't verify that the existing `main.py` or `layout.tsx` in the repository will be updated to reference them.

### Gap 2: Cross-Language Field Name Mismatch
**Bugs**: #555, #556, #558, #559 (4 bugs — 2×P0, 2×P1)
**Failure layer**: Architecture + Code Generation

Python models use `snake_case`. The architecture for #521 (Step 4) explicitly specified:
> "Types are defined as interface declarations that mirror the backend Python dataclasses' `to_dict()` output (snake_case field names)"

Yet the generated TypeScript code used `camelCase`. This is a **two-layer failure**:

| Bug | Symptom | Root cause |
|-----|---------|-----------|
| #555 | Frontend sends `uid`, `displayName` instead of `login`, `name` | Code gen ignored architecture's field name spec |
| #556 | 5 request payloads use camelCase, 5 response unwraps wrong | Code gen used TS conventions over architecture spec |
| #558 | TypeScript `CandidateStage` missing `RESURFACED` value | #520 added it to Python; #521 never learned about it |
| #559 | `GitHubProfile` and `Education` types never defined in TS | Architecture referenced them but no module generated them |

**Pipeline failure points**:

1. **`code_generator_main.py` (Lines 1059–1104)**: Post-generation validation ONLY checks JSON schema validity and architecture interface types. It does **not** verify:
   - Field naming convention compliance
   - Completeness of fields vs. specification
   - Cross-file naming consistency

2. **`agentic_arch_step9_completeness_LLM.prompt`**: Checks dependency graph and PRD coverage but has **no cross-language field name audit**. There is no `step9b_cross_audit` — this file does not exist in the codebase.

3. **Sub-issue isolation**: #520 (Nurture) added `RESURFACED` to Python's `CandidateStage` enum and 8 new fields to `RecruitingCandidate`. #521 (Frontend) was designed independently with only 12 enum values. The sub-issues share **no state** — each runs its own 12-step pipeline in isolation.

### Gap 3: Tests Generated Without Execution or Compilation
**Bugs**: #563, #564, #565 (3 bugs — all P2)
**Failure layer**: Code Generation

| Bug | Symptom | Root cause |
|-----|---------|-----------|
| #563 | `SyntaxError` — stray triple-quote at EOF, unclosed `textwrap.dedent()` | LLM generated malformed Python |
| #564 | Tests expect camelCase fields but models use snake_case; missing enum values | Tests generated from architecture descriptions, not actual code |
| #565 | 52/120 frontend tests fail — field mismatches, duplicate mock declarations | Same camelCase/snake_case drift in test expectations |

**Pipeline failure points**:

1. **`generate_test_LLM.prompt`**: The prompt says "Analyze ACTUAL code provided (exact function names, parameters, behavior)" — but during architecture generation (Steps 1–12), the actual code **doesn't exist yet**. Tests are generated from architecture descriptions, not real code.

2. **`agentic_architecture_orchestrator.py` (Lines 476–639)**: The validation loop runs Steps 9, 10, 11 with retry/fix logic, but:
   - Step 9 validates architecture structure via `jq` queries on `architecture.json`
   - Step 10 runs `pdd sync --dry-run` to validate config
   - Step 11 validates prompt dependencies
   - **None** of these run `python -c "import ast; ast.parse(...)"` or `tsc --noEmit`

3. **`pdd sync`** (runtime) does run tests, but `pdd arch` (design time) does not. Tests are already broken before `pdd sync` ever sees them.

### Gap 4: No Existing Codebase Auth/Guard Awareness
**Bugs**: #553, #554 (2 bugs — both P0)
**Failure layer**: Architecture

| Bug | Symptom | Root cause |
|-----|---------|-----------|
| #553 | Infinite refresh loop on `/recruit` page | `useCallback`/`useEffect` dependency cycle — missing `useRef` guard pattern |
| #554 | `/recruit` users redirected to `/interest` (waitlist) | `AuthContext.fetchUserStatus()` has redirect logic PDD didn't know about |

**Pipeline failure points**:

1. **`agentic_arch_step2_analyze_LLM.prompt`**: Performs "deep PRD analysis" to identify module boundaries and tech stack. It scans:
   - Feature decomposition into functional units
   - Shared concerns (auth, error handling, logging)
   - Tech stack determination

   But it does **not** scan for **runtime behavior** of existing code — auth redirects, middleware guards, React effect patterns, or route protection logic. It sees `AuthContext.tsx` as a file but doesn't analyze that `fetchUserStatus()` redirects unauthorized paths to `/interest`.

2. **`agentic_arch_step4_design_LLM.prompt`**: Designs module breakdown with dependency graph, but does not instruct the LLM to:
   - Identify which existing guards/middleware might intercept new routes
   - Check for React anti-patterns (effect dependency cycles)
   - Scan for cross-cutting redirect logic

### Gap 5: No Bidirectional Data Model Extensions
**Bugs**: #557 (1 bug — P1)
**Failure layer**: Architecture

| Bug | Symptom | Root cause |
|-----|---------|-----------|
| #557 | `RecruitingCandidate` missing 8 nurture fields; `from_dict()` → `to_dict()` round-trips strip them | Nurture modules (#520) ADD fields, but base model (#519) was generated first without them |

**Pipeline failure point** (`architecture_json.prompt`):

The architecture schema only has `dependencies` (A depends on B) — there is no `extends` or `augments` concept. When module B needs to add fields to module A's data model:
- The dependency arrow points B → A (B depends on A)
- But the data model change flows A ← B (A must include B's fields)
- PDD has no mechanism to represent or merge this reverse data flow

Since #519 (Core Backend) was designed independently from #520 (Nurture), the base `RecruitingCandidate` model never learned about nurture's 8 additional fields.

### Gap 6: No Dev Experience Module Generation
**Bugs**: #566, #567 (2 bugs — both P2)
**Failure layer**: Architecture

| Bug | Symptom | Root cause |
|-----|---------|-----------|
| #566 | No dev sign-in form for Firebase Auth emulator testing | PDD generates production features only |
| #567 | `credit_helpers.py` has `require_credits()` but no `grant_credits()` | Utility referenced in PRD but not extracted as a module |

**Pipeline failure point** (`agentic_arch_step1_analyze_prd_LLM.prompt`):

Step 1 extracts from the PRD:

- Non-functional requirements

### Gap 7: Sub-Issue Isolation — No Cross-Issue State Sharing
**Bugs**: #555, #556, #557, #558, #559 (5 bugs — overlaps with Gaps 2 & 5)
**Failure layer**: Architecture (Orchestrator)

When PRD #516 was split into 3 sub-issues, each ran its own independent 12-step pipeline. There is **no mechanism** for:
- Sub-issue B (#520) to share new fields/enums with sub-issue C (#521)
- Cross-sub-issue type validation
- Post-completion reconciliation

**Pipeline failure point** (`agentic_architecture_orchestrator.py`):

The orchestrator (Lines 277–410) processes **one issue at a time**. The Step 3 complexity analysis splits a large PRD into sub-issues, but:
- Each sub-issue gets its own `architecture.json`
- Each sub-issue runs Steps 4–12 independently
- Steps 9–11 validate **within** a single sub-issue only
- After all sub-issues complete, the parent issue (#516) tracks completion but does **not** run any cross-sub-issue consistency checks

**Concrete example**: #520's architecture added `CandidateStage.RESURFACED` and 8 new fields to `RecruitingCandidate` in Python. #521's architecture independently defined `CandidateStage` in TypeScript with only 12 values. No reconciliation step detected this drift.

### Fix 1: Entry Point Wiring System (P0)

**Problem**: PDD generates modules as standalone files but never patches existing entry points (`main.py`, `layout.tsx`, `__init__.py`).

**Fix A — Schema change** (`pdd/pdd/templates/architecture/architecture_json.prompt`):

Add an `entryPoints` field to the module schema:

```json
{
  "filename": "recruiting_register_Python.prompt",
  "filepath": "backend/functions/recruiting/recruiting_register.py",
  "entryPoints": [
    {
      "file": "backend/functions/main.py",
      "action": "append_registration",
      "description": "Add @https_fn.on_request() registration for recruitingRegister"
    }
  ]
}
```

**Fix B — Step 9 enhancement** (`pdd/prompts/agentic_arch_step9_completeness_LLM.prompt`):

Add to the validation checks:

```
ENTRY POINT WIRING CHECK:
For each module that exports a callable (Cloud Function, API route, page component):
  1. VERIFY it has an entryPoints entry in architecture.json
     OR is imported by another module that has one
  2. FLAG "orphan module" if nothing wires it into the application
  3. For web apps: verify all new routes have navigation links
     in existing layout/sidebar components
  4. For backends: verify all new endpoints are registered
     in the main entry point file
```

**Fix C — Step 4 enhancement** (`pdd/prompts/agentic_arch_step4_design_LLM.prompt`):

Add to the module design instructions:

```
For each new module, identify INTEGRATION POINTS:
  - Which existing entry point file must be updated (e.g., main.py, layout.tsx)
  - What registration/import/route code must be added
  - Whether a new index page or redirect is needed for nested routes
  - Which navigation components need new links
  - Create explicit "wiring" modules if the patches are non-trivial
```

### Fix 3: Sub-Issue Reconciliation Step (P0)

**Problem**: When PRD #516 was split into #519/#520/#521, each ran independently. #520 added `RESURFACED` to `CandidateStage` but #521 never learned about it.

**Fix A — Orchestrator change** (`pdd/pdd/agentic_architecture_orchestrator.py`):

After all sub-issues for a parent issue complete, add a reconciliation step:

```python
def reconcile_sub_issues(parent_issue_number: int, sub_issues: List[int]):
    """
    After all sub-issues complete, detect cross-sub-issue drift.
    """
    # 1. Collect all architecture.json files from sub-issues
    architectures = {}
    for sub_issue in sub_issues:
        arch_path = get_architecture_path(sub_issue)
        architectures[sub_issue] = load_architecture(arch_path)

    # 2. Build unified type registry
    type_registry = build_cross_issue_type_registry(architectures)

    # 3. Run cross-issue reconciliation LLM
    reconciliation_prompt = render_template(
        "cross_issue_reconcile_LLM.prompt",
        type_registry=type_registry,
        architectures=architectures
    )
    result = run_agentic_task(reconciliation_prompt)

    # 4. If drift detected, generate patch prompts
    if not _check_validation_result(result):
        patches = extract_patches(result)
        apply_reconciliation_patches(patches)
```

**Fix B — New prompt** (`pdd/prompts/cross_issue_reconcile_LLM.prompt`):

```
You are a cross-module consistency auditor. Given architecture.json files
from N sub-issues of the same parent PRD:

1. BUILD unified type registry across all sub-issues:
   - List every data model / type / enum defined in each sub-issue
   - Identify types that appear in multiple sub-issues

2. For each SHARED TYPE (appears in 2+ sub-issues):
   - Compare field lists → flag fields present in one but missing in another
   - Compare enum values → flag values present in one but missing in another
   - Compare type names → flag inconsistent naming

3. For each CROSS-LANGUAGE PAIR (e.g., Python dataclass ↔ TypeScript interface):
   - Verify field-for-field mapping exists
   - Check naming convention consistency (document the convention used)
   - Verify all enum values match

4. Output RECONCILIATION_RESULT: CONSISTENT or list of patches needed:
   - Which sub-issue's architecture.json needs updating
   - Which module's prompt needs additional fields/enums
   - Exact fields/values to add
```

### Fix 5: Test Compilation Validation (P1)

**Problem**: Generated tests have syntax errors (#563) and wrong field/enum expectations (#564, #565). Tests are validated by the architecture's structural checks (Steps 9–11) but never compiled or executed.

**Fix — Orchestrator change** (`pdd/pdd/agentic_architecture_orchestrator.py`):

After Step 8 generates prompt files (including test prompts), add a lightweight syntax check before Step 9:

```python
def validate_generated_test_syntax(generated_files: List[str], cwd: Path):
    """
    Quick syntax check on generated test files.
    Runs BEFORE Step 9 to catch obvious LLM generation errors.
    """
    errors = []
    for filepath in generated_files:
        full_path = cwd / filepath
        if not full_path.exists():
            continue

        if filepath.endswith('.py'):
            # Python: AST parse check
            result = subprocess.run(
                ['python', '-c', f'import ast; ast.parse(open("{full_path}").read())'],
                capture_output=True, text=True
            )
            if result.returncode != 0:
                errors.append(f"Python syntax error in {filepath}: {result.stderr}")

        elif filepath.endswith('.ts') or filepath.endswith('.tsx'):
            # TypeScript: tsc --noEmit check (if available)
            result = subprocess.run(
                ['npx', 'tsc', '--noEmit', '--allowJs', str(full_path)],
                capture_output=True, text=True, timeout=30
            )
            if result.returncode != 0:
                errors.append(f"TypeScript error in {filepath}: {result.stderr}")

    return errors
```

Note: This is a **syntax-only** check. Full test execution happens later during `pdd sync`. But catching `SyntaxError` at architecture time saves a round-trip.

### Fix 6: Existing Codebase Guard Scanning (P1)

**Problem**: PDD added `/recruit` pages without knowing that `AuthContext.fetchUserStatus()` redirects users to `/interest`. It also generated a `useCallback`/`useEffect` pattern that causes infinite loops.

```
CROSS-CUTTING CONCERN SCAN:
Scan the existing codebase for patterns that will affect new modules:

1. AUTH GUARDS & MIDDLEWARE:
   - Find files matching: AuthContext*, middleware*, auth_helpers*, _middleware*
   - For each, identify: which routes are protected, where unauthorized users are redirected
   - For each new route in the PRD, verify it won't be intercepted by existing guards

2. ROUTE PROTECTION:
   - Identify redirect logic in layout files, context providers, and middleware
   - Map: path pattern → redirect destination → condition

3. REACT EFFECT PATTERNS (for frontend modules):
   - Scan for useEffect/useCallback dependency patterns in existing layout/context files
   - Identify shared state that new components will inherit
   - Flag potential infinite render loops from missing ref guards

Output a "Cross-Cutting Concerns" section listing each guard/redirect
and which new modules need to account for it.
```

**Fix B — Step 4 enhancement** (`pdd/prompts/agentic_arch_step4_design_LLM.prompt`):

Add to module design:

```
For each new page/route module:
  - Check the Cross-Cutting Concerns from Step 2
  - If the route will be intercepted by an existing auth guard:
    create a module to update the guard with a bypass for the new route
  - If the module uses shared context (AuthContext, etc.):
    add the context file as a dependency and note required modifications
```

## 8. Broader Lessons

### 8.1 The "Orphan Module" Problem
PDD's 1-prompt-per-module design is powerful for isolation but creates **orphan modules** — files that exist but aren't wired into the application. Every non-trivial application has shared entry points (main.py, layout.tsx, router config) that must be patched when new modules are added. PDD needs a first-class concept for "patch existing file" alongside "create new file."

### 8.2 The "Sub-Issue Isolation" Problem
Splitting large PRDs into sub-issues is correct for complexity management, but the current implementation runs each sub-issue in a sealed pipeline. This works when sub-issues are truly independent (e.g., backend + unrelated frontend). It fails catastrophically when sub-issues share data models — which is the common case for full-stack features.

### 8.3 The Code Generator Was Faithful — The Prompts Were Wrong
We verified that the original generated code (which was never committed separately — `pdd fix` is the last step of `pdd sync`) was **faithful to the prompts**. The TypeScript types file used camelCase exactly as the prompt instructed. The bug wasn't the code generator ignoring the spec — it was the spec being wrong. This means fixes should target the **prompt generation pipeline** (Steps 4 and 8), not `code_generator_main.py`.

### 8.4 The "Architecture-Time Test" Problem
Tests generated during architecture design are aspirational — they describe what the code *should* do based on the spec. But since the actual code doesn't exist yet, tests are based on architecture descriptions, not real function signatures. This creates a systematic mismatch that only surfaces when tests are actually run (during `pdd sync`), which is too late to fix cheaply.

## 9. Code Faithfulness Verification & Bug-by-Bug Prompt Analysis

We examined all 20 recruiting prompts and compared them against the 16 bugs. The original generated code was never committed separately (the working tree files are `pdd sync` output including `pdd fix` as its final step), but by cross-referencing the prompt specifications against the bug descriptions, we can definitively determine whether each bug was caused by a prompt defect or a code generation error.

**Result: 14 of 16 bugs originated from prompt defects.** The code generator faithfully implemented what the prompts specified — the prompts themselves were wrong.

### 9.1 The Naming Convention Conflict Across Prompts

The single most damaging issue is an **internal contradiction** across the prompt set:

| Prompt | Convention | What it says |
|--------|-----------|-------------|
| `recruiting_models_Python` | **snake_case** | All fields snake_case for Firestore |
| `recruiting_register_Python` | **snake_case** | Request/response: `candidate_uid`, `conversation_id` |
| `recruiting_chat_Python` | **snake_case** | Response: `conversation_status`, `signals_extracted` |
| `recruiting_admin_api_Python` | **snake_case** | All 7 endpoints use snake_case |
| `recruiting_types_TypeScript` | **camelCase** | "Python backend serializes snake_case fields to camelCase in JSON responses" |
| `recruiting_api_client_TypeScript` | **CONFLICTING** | Dependency reference shows snake_case (`conversation_status`), but imports camelCase types |

Every backend prompt says snake_case. The types prompt says camelCase based on a **phantom serialization layer** that doesn't exist. The api_client prompt is internally contradictory — its dependency reference section shows snake_case response fields but it imports camelCase types.

The code generator faithfully followed each prompt. Backend code produced snake_case JSON. Frontend code consumed camelCase fields. The mismatch was baked into the prompts from the start.

### 9.2 Bug-by-Bug Prompt Verdict

#### #552 — Cloud Functions not registered in main.py
**Prompt bug: YES — omission.**

No backend prompt mentions `main.py` registration. The prompts define Cloud Functions (`recruitingRegister`, `recruitingChat`, etc.) and specify `@https_fn.on_request()` decorators, but never say "register this function in the entry point." PDD generated a sidebar update prompt (`admin_recruiting_sidebar_update`) that explicitly patches `layout.tsx` — so PDD knows how to do wiring — but didn't generate equivalent wiring for the backend.

#### #553 — Infinite refresh loop on /recruit
**Prompt bug: PARTIAL — missing defensive pattern.**

The `recruit_landing_page` prompt says "auto-redirect if authenticated" but doesn't specify a `useRef` guard to prevent re-render loops. The `useCallback`/`useEffect` dependency cycle that caused infinite loops is a React anti-pattern the code generator produced. The prompt could have prevented this by specifying "use a ref guard to ensure registration runs only once."

#### #554 — AuthContext redirects /recruit to /interest
**Prompt bug: YES — omission of existing codebase behavior.**

The landing page prompt uses `useAuth()` from `@/context/AuthContext` but doesn't document that `fetchUserStatus()` redirects users without a waitlist entry to `/interest`. The prompt has no way to know about this cross-cutting concern. Step 2 (deep analysis) should have scanned `AuthContext.tsx` for redirect behavior and flagged it.

#### #555 — GitHubProfile sends wrong field names
**Prompt bug: YES — phantom serialization.**

The `recruiting_types` prompt (line 155) says *"Python backend serializes snake_case fields to camelCase in JSON responses."* Based on this, the types file defines `GitHubProfile` with camelCase field expectations. The landing page prompt constructs the GitHub profile from Firebase Auth properties (`user.displayName`, `user.photoURL`) rather than mapping to the backend's expected fields (`login`, `name`, `avatar_url`). The api_client prompt's dependency reference (line 94) correctly shows `{ login: string; name: string; email: string; avatar_url: string }` (snake_case), but this conflicts with the camelCase types file it imports.

#### #556 — API request/response field name mismatches (camelCase vs snake_case)
**Prompt bug: YES — the core prompt defect.**

This is the direct consequence of `recruiting_types_TypeScript.prompt` line 155: *"Python backend serializes snake_case fields to camelCase in JSON responses (the frontend receives camelCase)."* This statement is false. The Python backend uses `jsonify(model.to_dict())` which preserves snake_case. No `snake_to_camel()` conversion exists anywhere. The code generator faithfully produced camelCase request payloads and camelCase response field access — exactly what the prompt specified — causing 5 request mismatches and 5 response unwrapping failures.

#### #557 — RecruitingCandidate missing 8 nurture extension fields
**Prompt bug: YES — cross-sub-issue omission.**

The `recruiting_models` prompt (sub-issue #519) lists 28 fields and 12 enum values for `RecruitingCandidate`. The `recruiting_nurture_models` prompt (sub-issue #520) documents 8 extension fields and says they are "defined in `recruiting/recruiting_models.py`." But since #519 and #520 were separate sub-issues, the models prompt was generated without knowledge of the nurture extensions. The nurture prompt **assumed** the base model would include them — nobody told the base model prompt.

#### #558 — Missing RESURFACED in TypeScript CandidateStage
**Prompt bug: YES — cross-sub-issue enum drift.**

The `recruiting_types` prompt (line 33) lists 12 `CandidateStage` values — no `RESURFACED`. The `recruiting_models` prompt also lists only 12. The `RESURFACED` value was added by the nurture sub-issue (#520) but never propagated to the frontend sub-issue (#521). Each sub-issue ran its own pipeline in isolation.

#### #559 — Missing GitHubProfile and Education TypeScript types
**Prompt bug: YES — omission from types prompt.**

The `recruiting_types` prompt defines 16 types in its interface section (lines 2-22). `GitHubProfile` and `Education` are **not in the list**. However, the `recruiting_api_client` prompt (lines 94-95) references them: `GitHubProfile — { login, name, email, avatar_url }` and `Education — { school, gpa?, major?, graduation_year? }`. The api_client prompt assumed the types file would define them, but the types prompt never specified them. The code generator faithfully omitted them from the types file.

#### #560 — /admin/recruiting returns 404 (no index page)
**Prompt bug: YES — omission.**

The `admin_recruiting_pipeline_page` prompt places itself at `/admin/recruiting/pipeline`. The `admin_recruiting_settings_page` prompt places itself at `/admin/recruiting/settings`. No prompt creates an index page at `/admin/recruiting/page.tsx` to handle the bare `/admin/recruiting` route. A redirect page was needed but never specified.

#### #561 — Admin sidebar missing recruiting nav links
**Prompt bug: NO — prompt exists and is correct.**

The `admin_recruiting_sidebar_update` prompt correctly specifies adding two nav items (`Recruiting` and `Recruit Settings`) to `frontend/src/app/admin/layout.tsx`. This is one of the few prompts that addresses wiring. If the code wasn't generated or applied, that's a pipeline execution issue, not a prompt defect.

#### #562 — Duplicate imports in recruiting/__init__.py
**Prompt bug: YES — omission, no coordination.**

No prompt manages the `recruiting/__init__.py` file. Multiple modules were generated expecting their own imports, and without a single prompt coordinating the init file, duplicates were introduced.

#### #563 — test_recruiting_core.py has syntax errors
**Prompt bug: NO — LLM code generation error.**

The `recruiting_core_tests` prompt is well-specified with correct snake_case fields and detailed test scenarios. The syntax errors (stray triple-quote at EOF, unclosed `textwrap.dedent()` calls) are pure LLM generation errors — the prompt didn't cause them.

#### #564 — Tests expect wrong enum/field names
**Prompt bug: YES — partial, cross-sub-issue.**

The `recruiting_core_tests` prompt uses snake_case field names (correct). But it references the `recruiting_models` prompt which has only 12 `CandidateStage` values (missing `RESURFACED`). Tests generated from this prompt would assert only 12 enum values. Additionally, since the types prompt specified camelCase, any tests touching frontend-facing code would use camelCase mock data — mismatching the actual snake_case backend.

#### #565 — Frontend Jest tests fail (52 failures)
**Prompt bug: YES — indirect, follows wrong convention.**

The `recruiting_frontend_tests` prompt doesn't explicitly specify field naming. It references types from `@/types/recruiting` — which was generated with camelCase per the types prompt. So test mock data would use camelCase fields like `githubUsername`, `avatarUrl`, while the actual (fixed) code uses `github_username`, `avatar_url`. The phantom camelCase assumption cascaded into test expectations.

#### #566 — No dev-mode sign-in for emulator testing
**Prompt bug: YES — omission.**

No prompt mentions dev-mode authentication, Firebase Auth emulator compatibility, or test sign-in forms. The PRD analysis (Step 1) doesn't extract development requirements. PDD generates production features only.

#### #567 — Missing credit_helpers.grant_credits utility
**Prompt bug: YES — assumed utility exists.**

The `recruiting_send_homework` prompt says "Grant the configured number of credits" and references `credit_helpers`. But no prompt specifies creating a `grant_credits()` function in `credit_helpers.py`. The prompt assumed the function already existed in the codebase — it didn't.

## 10. Proposed Solution: Shared Context Documents

The fundamental problem is that PDD's 1-prompt-per-module architecture creates **information silos**. Each prompt is self-contained, so cross-cutting concerns (field names, auth behavior, entry points) are either duplicated inconsistently or missing entirely.

### 10.1 The Concept: "Context Documents"

When PDD splits a PRD into sub-issues or generates prompts for a multi-module project, it should first generate **shared context documents** that all prompts reference. These are not code — they are structured reference files that establish contracts.

```
prompts/
  _context/                          ← NEW: shared context directory
    data_dictionary.yaml             ← canonical field names, types, enums
    api_contracts.yaml               ← request/response schemas per endpoint
    integration_points.yaml          ← entry points, nav links, guards
    existing_codebase_contracts.yaml ← auth behavior, middleware, redirects
    env_vars.yaml                    ← all environment variables
  recruiting_models_Python.prompt
  recruiting_types_TypeScript.prompt  ← references _context/data_dictionary.yaml
  recruiting_register_Python.prompt   ← references _context/integration_points.yaml
  ...
```

### 10.2 Data Dictionary (`_context/data_dictionary.yaml`)

A single source of truth for all data models across languages:

```yaml
# prompts/_context/data_dictionary.yaml
# Generated by pdd arch Step 4, updated by all sub-issues

models:
  RecruitingCandidate:
    source_module: recruiting_models_Python.prompt
    source_sub_issue: "#519"
    serialization: snake_case  # ← Python .to_dict() uses snake_case, NO conversion
    fields:
      - name: uid
        python_type: str
        ts_type: string
        constraints: "primary key, auto-generated UUID"
      - name: github_username
        python_type: Optional[str]
        ts_type: string | null
        constraints: ""
      - name: candidate_stage
        python_type: CandidateStage
        ts_type: CandidateStage
        constraints: "enum value"
      # ... all 35+ fields listed once
    extensions:
      - source_module: recruiting_nurture_models_Python.prompt
        source_sub_issue: "#520"
        fields:
          - name: crm_issue_number
            python_type: Optional[int]
            ts_type: number | null
          - name: nurtured_at
            python_type: Optional[datetime]
            ts_type: string | null  # ISO 8601
          # ... 8 nurture fields

enums:
  CandidateStage:
    source_module: recruiting_models_Python.prompt
    values:
      - name: APPLIED
        python_value: "applied"
        ts_value: "'applied'"
      - name: SCREENED
        python_value: "screened"
        ts_value: "'screened'"
      # ... all 13 values including RESURFACED
    extensions:
      - source_module: recruiting_nurture_models_Python.prompt
        source_sub_issue: "#520"
        added_values:
          - name: RESURFACED
            python_value: "resurfaced"
            ts_value: "'resurfaced'"

naming_convention:
  python: snake_case
  typescript: snake_case  # matches JSON from backend — NO camelCase conversion
  reason: "Backend uses jsonify(model.to_dict()) with no case transformation"
```

### 10.5 Existing Codebase Contracts (`_context/existing_codebase_contracts.yaml`)

Documents runtime behaviors that new modules must account for:

```yaml
# prompts/_context/existing_codebase_contracts.yaml

auth_guards:
  - file: frontend/src/context/AuthContext.tsx
    behavior: |
      fetchUserStatus() checks if user has a waitlist entry.
      If no entry found AND path is not in bypass list, redirects to /interest.
    bypass_paths:
      - /interest
      - /recruit    # ← MUST be added for recruiting to work
      - /recruit/*
    effect_pattern: |
      Uses useEffect with fetchUserStatus dependency.
      New pages using this context MUST use useRef guard to prevent
      infinite re-render when effect triggers state updates.

  - file: frontend/src/app/admin/layout.tsx
    behavior: |
      Wraps all /admin/* routes. Checks isAdmin flag.
      Non-admin users see "Access Denied" page.

  - file: backend/functions/utils/auth_helpers.py

      - name: require_authentication
        behavior: "Validates Firebase ID token from Authorization header"
      - name: require_admin
        behavior: "Validates token AND checks admin flag in Firestore"
```

### 10.6 How Context Documents Flow Through the Pipeline

```
Step 1: Analyze PRD
  ↓
Step 2: Deep Analysis (scans existing codebase)
  ↓
Step 3: Split into sub-issues
  ↓
Step 4: Design modules → GENERATES _context/ documents     ← NEW
  ↓                        (data_dictionary.yaml,
  ↓                         api_contracts.yaml,
  ↓                         integration_points.yaml,
  ↓                         existing_codebase_contracts.yaml)
  ↓
Step 5-7: Research, scaffolding, config
  ↓
Step 8: Generate prompts → Each prompt <include>s relevant    ← ENHANCED
  ↓                         context documents
  ↓
Step 9: Completeness check → VALIDATES prompts against        ← ENHANCED
  ↓                           context documents
  ↓
Step 10-11: Sync and dependency validation
  ↓
[Sub-issue completes]
  ↓
Step 13 (NEW): Cross-issue reconciliation                     ← NEW
  → Merges _context/ documents from all sub-issues
  → Detects drift (missing fields, enums, naming mismatches)
  → Patches prompts to resolve drift
```

### 10.7 Key Benefits

| Problem | Without Context Docs | With Context Docs |
|---------|---------------------|-------------------|
| Field naming drift | Each prompt decides independently | All prompts reference `data_dictionary.yaml` |
| Missing enum values | Sub-issues don't share enums | `data_dictionary.yaml` has canonical enum with extensions |
| Phantom serialization | Prompt assumes camelCase conversion | `naming_convention` field says "snake_case, no conversion" |
| Orphan modules | No one wires entry points | `integration_points.yaml` lists all registrations |
| Auth guard conflicts | Prompts don't know about redirects | `existing_codebase_contracts.yaml` documents guards |
| Cross-sub-issue drift | No shared state | `_context/` is shared across sub-issues and merged |

### 10.8 Implementation Priority

1. **Start with `data_dictionary.yaml`** — This alone would have prevented 8 of 16 bugs (#555, #556, #557, #558, #559, #564, #565, and partially #563). It's the single highest-value addition.

2. **Add `integration_points.yaml`** — Prevents 4 more bugs (#552, #560, #561, #562).

3. **Add `existing_codebase_contracts.yaml`** — Prevents 2 more bugs (#553, #554).

4. **Add `api_contracts.yaml`** — Reinforces `data_dictionary.yaml` for API-specific field mapping.

**Combined with pipeline fixes from Section 5, this would prevent all 16 bugs.**
