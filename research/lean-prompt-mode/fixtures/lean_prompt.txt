% You are an expert Python engineer. Implement the validate_upload module.

<pdd-reason>Validates file uploads before writing to storage: checks type, size, and duplicate hash.</pdd-reason>

<pdd-interface>

  "module": {

}
</pdd-interface>

% Vocabulary

- Allowed type: a MIME type present in `config.allowed_mime_types`.
- File size: the byte length of the raw file content before any transformation.
- Duplicate upload: an upload whose normalized SHA-256 hash and tenant ID match a previously accepted upload record in the store.
- Normalized hash: SHA-256 of the file content with no leading/trailing whitespace normalization (binary files are hashed as-is).

<responsibility>
Validate file uploads: MIME type, size limit, and duplicate detection. Return a structured ValidationResult.
</responsibility>

<non_responsibilities>
- Does not authenticate the caller.
- Does not authorize tenant-level quotas or billing checks.
- Does not write to storage; validation only.
- Does not send notifications.
- Does not perform virus scanning.
</non_responsibilities>

<contract_rules>
R1 - Type allowlist
For every upload, the system MUST reject the request when the file's detected MIME type is not in config.allowed_mime_types.
This rule is violated if a file with a disallowed MIME type reaches the storage write path.

R2 - Size limit
For every upload, the system MUST reject the request when the file size in bytes exceeds config.max_file_bytes.
This rule is violated if an over-limit file is accepted.

R3 - Duplicate rejection
For every upload, the system MUST reject the request when the normalized SHA-256 hash and tenant_id match an existing accepted record.
This rule is violated if a duplicate file is accepted a second time.

R4 - No storage write before validation
The system MUST NOT call the storage write path for any request rejected by R1, R2, or R3.
This rule is violated if a rejected upload triggers a storage write.

R5 - Structured result
For every upload, the system MUST return a ValidationResult with is_valid (bool), error_code (str or None), and error_detail (str or None).
This rule is violated if the function raises an unhandled exception for expected invalid inputs.
</contract_rules>

<capabilities>
- MAY read file content (bytes) from the provided file object.
- MAY read duplicate records from the provided store interface.
- MAY compute SHA-256 hash of file content.
- MUST NOT write to storage.
- MUST NOT send emails or notifications.
- MUST NOT read environment variables not provided via config.
- MUST NOT log raw file content or tenant secrets.
</capabilities>

% Requirements

1. Function: `validate_upload(file_obj, tenant_id, config) -> ValidationResult`
2. `file_obj` exposes `.read() -> bytes` and `.content_type: str`.
3. `config` exposes `.allowed_mime_types: list[str]` and `.max_file_bytes: int`.
4. Reject (R1) if `file_obj.content_type` not in `config.allowed_mime_types`.
5. Reject (R2) if `len(file_obj.read()) > config.max_file_bytes`.
6. Reject (R3) if `store.exists(sha256_hash, tenant_id)` returns True.
7. Accept: return `ValidationResult(is_valid=True, error_code=None, error_detail=None)`.
8. Reject: return `ValidationResult(is_valid=False, error_code=<CODE>, error_detail=<MSG>)`.
9. Error codes: `"TYPE_NOT_ALLOWED"`, `"FILE_TOO_LARGE"`, `"DUPLICATE_UPLOAD"`.
10. Do not raise exceptions for expected invalid inputs; only for programming errors (e.g., missing required config fields).
