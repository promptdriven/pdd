# Epic: Repo-Bloat / Context-Window Experiment — v2 revision (#1209)

**Issue:** [#1209](https://github.com/promptdriven/pdd/issues/1209) — Research: measure agentic localization degradation under repository bloat
**Epic branch:** `epic/1209-repo-bloat-v2`
**Builds on:** the merged pilot design (PRs [#1401](https://github.com/promptdriven/pdd/pull/1401), [#1419](https://github.com/promptdriven/pdd/pull/1419)) and the research-context work (PR [#1569](https://github.com/promptdriven/pdd/pull/1569)).

This epic turns the v1 *design document* into a **runnable experiment plus a
paper draft**, incorporating a round of review feedback. It changes the design
where the feedback requires it, and implements the harness pieces the v1
design only specified.

## The feedback being incorporated

1. **Exclude SWE-bench from the core experiment.**
2. **Study context snapshots** — do not rely only on file-level (filesystem
   read) investigation.
3. **Investigate whether token bloat happens gradually across agent
   iterations** (within a run), not only across repo sizes.
4. **Consider forking Codex or using PDD as the target repo**, then observe
   context-window behavior while the CLI works.
5. **Define "distractor file" precisely, generate many distractors from the
   definition/template, scale aggressively (50x), and observe where
   context-window performance starts to break down.**
6. **Measure how well the instrumented CLI performs its assigned tasks as
   distractors increase.**

## Decisions made in this revision (summary — full rationale in `design.md` v2)

| # | Feedback | Decision |
|---|----------|----------|
| 1 | SWE-bench | Excluded as a scenario source or evaluation substrate for the core experiment (contaminated, repo size uncontrolled, task heterogeneity). Retained **only** as related work and as the origin of the hidden-test idea. |
| 2 | Context snapshots | New **primary instrument**: an API-boundary **recording proxy** that captures the byte-exact request payload (the composed context window) of every model request, plus the agent's own session logs. The FUSE/OverlayFS filesystem tap is **demoted to an optional cross-check tier** — file-level reads are no longer the primary evidence. |
| 3 | Iteration-level bloat | New pre-registered hypothesis **H2**: within a run, irrelevant-context share accumulates *gradually* across iterations (tool-result accretion) rather than arriving in one jump. Measured from the per-request snapshot series (early/middle/late run phases). |
| 4 | Fork Codex vs. PDD target | **Do not fork Codex** — instrument *around* the stock CLI at the API boundary (base-URL override) so external validity is preserved and the harness is agent-agnostic. **Use PDD itself as the primary target repository** (pinned commit, dependency-sliced cores, seeded bugs): a real mid-size production codebase we control end-to-end, with an auditable core/distractor boundary. |
| 5 | Distractor definition + scale | A formal, checkable definition of *distractor file* plus a deterministic generator with three modes — `regrow` (real out-of-core files), `mutate` (derived variants that extend the pool past the repo's own size), `template` (vocabulary-harvested synthetic modules) — filling token budgets on a ladder `1x/2x/5x/10x/20x/50x`, with a documented **breakdown probe** beyond 50x. |
| 6 | Task performance vs. scale | `hidden_pass_rate`, tokens/time-to-success, and failure classes reported **per size step**, so the report locates the knee of the degradation curve, not just its endpoints. |

## What "done" looks like for this epic

- The v2 design is internally consistent with the feedback and pre-registered
  before any model run.
- `harness/` contains runnable, tested scaffolding: distractor generation,
  variant materialization, context-snapshot capture, per-iteration analysis,
  and an end-to-end mock run producing a report — **no model tokens required
  to validate the pipeline**.
- The paper draft states the research question, definitions, methodology,
  metrics, expected results, and limitations, with SWE-bench appearing only in
  related work.
- Real pilot runs (with a live model + pinned Codex CLI) remain follow-up
  work; every blocker for them is enumerated, none is hidden.
