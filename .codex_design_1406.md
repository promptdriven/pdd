## What #1406 wants
One canonical final gate = Layer 1 (PR-scoped 8-step checkup, no new PR, same-branch push only, no broad churn) THEN Layer 2 (maintainer-style review-loop), exposed as `pdd checkup --pr <PR> --issue <ISSUE> --final-gate` and `run_agentic_checkup(..., final_gate=True)`, with pdd-issue final verification routed through it. The final gate returns a REAL ship verdict (from the review-loop's current-run final-state.json), unlike `--review-loop` whose success only means "report produced".

## The proposed design (in the brief)
- Library: add `final_gate` param to `run_agentic_checkup`; extract Layer-2 setup into `_run_pr_review_loop(...)` shared by `review_loop` and `final_gate`; `final_gate` runs Layer 1 (orchestrator) → if fail, skip Layer 2 → else Layer 2 → derive ship verdict via new pure predicate `_review_loop_ship_verdict(state, has_issue)` reading final-state.json (clearing any stale final-state BEFORE Layer 2). Reject combining `--final-gate` with `--review-loop`/`--no-fix`/`--review-only`/`--start-step`; require `--pr`+`--issue`. Opt-in; does NOT change plain `pdd checkup --pr --issue`.
- e2e: `_run_final_checkup_on_pr` passes `final_gate=True` + the cross-worktree SHA disambiguation fix (accept post-head if it equals the legacy checkup-worktree head OR the review-loop's verified_head_sha from final-state.json; only fail "external push" if NEITHER).
