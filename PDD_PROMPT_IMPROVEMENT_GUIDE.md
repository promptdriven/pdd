# PDD Prompt Improvement Guide

## Purpose

This document guides the implementation of three improvements to PDD's prompt generation pipeline, addressing the root cause of 14/16 bugs found in the recruiting app (see `PDD_GAP_ANALYSIS.md`).

The core insight: **the code generator was faithful to the prompts — the prompts were wrong.** Fixing the prompts fixes the bugs.

---

## Current State: How Prompt Generation Works Today

### The Problem: Single-Shot, All-At-Once Generation

Today, Step 8 of `pdd arch` generates **ALL prompt files in a single LLM call**:

```
architecture.json (42 modules)
        │
        ▼
   Step 8 LLM (single 900s agentic task)
        │
        ▼
  42 .prompt files written at once
```

The LLM receives the full `architecture.json` and writes every prompt file in one session. Each prompt is self-contained — there is no shared reference document. The LLM must independently derive field names, naming conventions, API contracts, and integration requirements for each prompt from the architecture description alone.

### Why This Causes Bugs

1. **No single source of truth**: The LLM derives field names from architecture descriptions. For the Python prompt it writes `github_username` (snake_case). For the TypeScript prompt it writes `githubUsername` (camelCase) and adds a false assumption that the backend converts between them.

2. **Cross-sub-issue blindness**: When a PRD is split into sub-issues (#519 backend, #520 nurture, #521 frontend), each sub-issue runs its own Step 8. Sub-issue #520 adds `RESURFACED` to an enum, but sub-issue #521's Step 8 never sees it.

3. **Context window pressure**: Generating 42 prompts in one LLM call means the model must hold the entire architecture in context while writing each prompt. By prompt #30, earlier decisions may be forgotten or contradicted.

4. **No verification against reality**: The LLM assumes conventions (like snake→camel serialization) without checking the actual codebase.

## Improvement 1: Shared Context Documents

### What

Before generating any module prompts, Step 8 should first generate a set of **shared context documents** that establish canonical contracts. Every module prompt then `<include>`s the relevant context documents instead of independently deriving conventions.

### New File Structure

```
prompts/
  _context/                              ← NEW: generated BEFORE module prompts
    data_dictionary.yaml                 ← canonical field names, types, enums
    api_contracts.yaml                   ← request/response schemas per endpoint
    integration_points.yaml              ← entry points, nav links, init files
    existing_codebase_contracts.yaml     ← auth guards, middleware, redirects
  recruiting_models_Python.prompt        ← <include>prompts/_context/data_dictionary.yaml</include>
  recruiting_types_TypeScript.prompt     ← <include>prompts/_context/data_dictionary.yaml</include>
  recruiting_register_Python.prompt      ← <include>prompts/_context/api_contracts.yaml</include>
  ...
```

## Improvement 2: Phased Prompt Generation (Not All At Once)

### What

Split Step 8 from a single monolithic LLM call into **multiple focused phases**, each generating a subset of prompts with the shared context documents available.

### Current Flow (Single Shot)

```
Step 8: Single LLM call → writes all 42 prompts → done
```

### Proposed Flow (Phased)

```
Step 8a: Generate shared context documents
    │    Input: architecture.json + existing codebase scan from Step 2
    │    Output: _context/data_dictionary.yaml
    │            _context/api_contracts.yaml
    │            _context/integration_points.yaml
    │            _context/existing_codebase_contracts.yaml
    │
    │    This is a SEPARATE LLM call focused ONLY on establishing contracts.
    │    It reads the actual codebase to verify conventions (e.g., does the
    │    backend do snake→camel conversion? No → document that).
    ▼
Step 8b: Generate foundational prompts (priority 1-3)
    │    Input: architecture.json + _context/* documents
    │    Output: data model prompts, config prompts, types prompts
    │
    │    These prompts <include> the context documents.
    │    Models and types are generated first because everything depends on them.
    ▼
Step 8c: Generate dependent prompts (priority 4+)
    │    Input: architecture.json + _context/* + prompts from 8b
    │    Output: API endpoints, pages, components, tests
    │
    │    These prompts <include> both context documents AND
    │    foundational prompts (via <include>examples/...).
    ▼
Step 8d: Generate integration prompts
         Input: _context/integration_points.yaml
         Output: main.py wiring prompt, __init__.py prompt,
                 sidebar update prompt, index redirect prompt
```

### Why Phased Generation Is Better

1. **Reduced context pressure**: Each LLM call handles ~10 prompts instead of 42. Less chance of internal contradictions.

2. **Context documents as ground truth**: By generating `_context/` first, every subsequent prompt generation call has a verified reference to `<include>`. The LLM can't invent a phantom serialization layer because `data_dictionary.yaml` explicitly says "no conversion."

3. **Foundational prompts available to dependents**: When generating `recruiting_register_Python.prompt` in Step 8c, the `recruiting_models_Python.prompt` already exists and can be read/referenced.

4. **Integration prompts last**: Entry point wiring is generated as an explicit phase after all modules are known. This prevents orphan modules.

### Implementation in the Orchestrator

Modify `agentic_architecture_orchestrator.py` Step 8 section (lines 411-475):

## Improvement 3: Teach `pdd generate` to Include Shared Context Documents

### What

Teach Step 8 (prompt generation) to embed XML `<include>` tags in every generated `.prompt` file so that when `pdd sync` later preprocesses those prompts, the code generator LLM automatically receives the shared context documents as part of its input.

**No changes to `pdd sync` or the preprocessor are needed.** PDD's `<include>` tag resolver (`pdd/pdd/preprocess.py`) already handles file paths relative to the working directory. Since `prompts/_context/data_dictionary.yaml` is a real file on disk, `<include>prompts/_context/data_dictionary.yaml</include>` will resolve correctly with the existing preprocessor.

The **only change** is in Step 8's prompt templates — the LLM that generates `.prompt` files must be instructed to add `<include>` tags pointing to the context documents.

### How It Works End-to-End

```
Step 8 (pdd generate)                    pdd sync
─────────────────────                    ────────

Step 8a: Generate _context/*.yaml
         (data_dictionary, api_contracts, etc.)
              │
              ▼
Step 8b/8c: Generate .prompt files
         LLM is instructed to embed
         <include> tags in each prompt
              │
              ▼
  recruiting_types_TypeScript.prompt     Preprocessor expands <include> tags
  contains:                              → LLM receives full YAML content
    <include>prompts/_context/           → Code matches data dictionary exactly
      data_dictionary.yaml</include>
```

### Step 8 Prompt Template Changes

Add to `agentic_arch_step8b_foundational_LLM.prompt` and `agentic_arch_step8c_dependent_LLM.prompt`:

```
CRITICAL INSTRUCTION — Shared Context Documents:

Every prompt you generate MUST include these XML <include> tags near the top,
BEFORE the main requirement paragraphs. These tags will be resolved by the
preprocessor at code generation time, injecting the shared context into
the code generator's input.

For ALL prompts (models, types, endpoints, pages, tests):
  <data_dictionary>
    <include>prompts/_context/data_dictionary.yaml</include>
  </data_dictionary>

For API endpoint prompts (backend functions and frontend API hooks):
  <api_contracts>
    <include>prompts/_context/api_contracts.yaml</include>
  </api_contracts>

For backend Cloud Function prompts:
  <integration_points>
    <include>prompts/_context/integration_points.yaml</include>
  </integration_points>

For frontend page/component prompts:
  <existing_codebase>
    <include>prompts/_context/existing_codebase_contracts.yaml</include>
  </existing_codebase>

After the <include> tags, add a CONSTRAINTS section in the prompt body:

CONSTRAINTS (from shared context documents):
- Field naming: Use the convention specified in data_dictionary.serialization.
- Serialization: Do NOT assume any automatic snake_case → camelCase conversion.
- All field names, types, and enum values MUST match data_dictionary exactly.
- All API request/response schemas MUST match api_contracts exactly.
```

### What the Generated Prompts Look Like

**Example — `recruiting_types_TypeScript.prompt` (generated by Step 8b):**

The types prompt becomes dramatically simpler — instead of manually listing 35+ fields with naming conventions, it references the data dictionary:

```
<pdd-reason>...</pdd-reason>

4. Use `as const` pattern for enums, `| null` for optional fields.
5. All datetime fields → string (ISO 8601).
```

This makes it **impossible** for the types prompt to invent a phantom serialization convention, because the convention is explicitly stated in the data dictionary that gets injected at code generation time.

### Why This Works Without Changing pdd sync

PDD's include resolution flow is:

1. `pdd sync` calls `preprocess.py` on each `.prompt` file
2. `preprocess.py` finds `<include>path</include>` tags
3. It resolves paths relative to CWD → package root → repo root
4. The tag is replaced with the file's content
5. The expanded prompt (with full YAML inlined) is sent to the code generator LLM

Since `prompts/_context/data_dictionary.yaml` is a real file generated by Step 8a and sitting on disk, the existing preprocessor resolves it automatically. **Zero code changes to `pdd sync` or `preprocess.py`.**

## Improvement 3b: Cross-Sub-Issue Context Document Merging

### The Problem

When a PRD is split into sub-issues (#519, #520, #521), each sub-issue runs its own Step 8 and generates its own `_context/` documents. Sub-issue #520 (nurture) adds `RESURFACED` to the enum and 8 new fields, but sub-issue #521 (frontend) never sees them.

### The Fix

After ALL sub-issues complete their Step 8, add a **reconciliation step** that merges all `_context/data_dictionary.yaml` files into one unified version, then re-validates all prompts against it.

```
Sub-issue #519 completes Step 8 → _context/data_dictionary.yaml (v1: 12 enum values, 28 fields)
Sub-issue #520 completes Step 8 → _context/data_dictionary.yaml (v2: 13 enum values, 36 fields)
Sub-issue #521 completes Step 8 → _context/data_dictionary.yaml (v3: 12 enum values, 28 fields — STALE)
                                          │
                                          ▼
                               Reconciliation Step
                                          │
                            Merge v1 + v2 + v3 → unified
                            Detect: v3 missing RESURFACED
                            Detect: v3 missing 8 nurture fields
                            Patch: update #521's prompts
                                          │
                                          ▼
                            _context/data_dictionary.yaml (final: 13 enum values, 36 fields)
                            All prompts updated to reference final version
```

### Implementation

Add to `agentic_architecture_orchestrator.py` after all sub-issues complete:

```python
def reconcile_context_documents(parent_issue: int, sub_issues: List[int]):
    """Merge _context/ from all sub-issues into unified version."""

    # 1. Collect all data dictionaries
    all_dicts = []
    for sub in sub_issues:
        dict_path = get_context_path(sub, "data_dictionary.yaml")
        if dict_path.exists():
            all_dicts.append(load_yaml(dict_path))

    # 2. Merge: union of all fields, union of all enum values
    merged = merge_data_dictionaries(all_dicts)

    # 3. Write unified version
    write_yaml("prompts/_context/data_dictionary.yaml", merged)

    # 4. Detect drift: which sub-issues have stale prompts?
    for sub in sub_issues:
        stale_prompts = find_stale_prompts(sub, merged)
        if stale_prompts:
            # Re-run Step 8c for this sub-issue with the merged context
            regenerate_prompts(sub, stale_prompts, merged)
```
